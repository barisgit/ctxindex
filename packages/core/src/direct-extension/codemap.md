# packages/core/src/direct-extension/

## Responsibility

Materializes package-backed Extension roots, records their immutable dependency resolution, and manages both direct lifecycle operations and Catalog-authoring/exact-replay installation while keeping startup offline.

## Design

- `target.ts` validates npm/Git/local targets, rejects embedded credentials, normalizes local origins, and exposes sanitized requested-target projections.
- `materializer.ts` invokes Bun in an isolated non-interactive environment with scripts disabled, preserving only credential-free HTTP(S) `BUN_CONFIG_REGISTRY` configuration. Fresh npm materialization derives the installed package name, exact version, and integrity; fresh Git materialization requires Bun to report a credential-free repository identity and exact commit, then rewrites the frozen workspace dependency from the mutable requested ref. After Bun operations in both fresh and exact paths, it normalizes the complete staged tree to deterministic directory/file modes (preserving only executable intent), rejects special files, and re-enforces `bun.lock` mode `0600`, making materialization hashes independent of source-root permissions. It first writes then freezes Bun 1.3.14 text lockfiles, synthesizes a canonical empty lock and skips no-op frozen installs for dependency-free local packages, canonicalizes and hashes a sanitized resolution envelope, and returns that artifact with every materialization. Exact replay restores the canonical manifest/lockfile, freezes production dependencies when required, and verifies npm package/version/integrity or Git repository/commit authority together with the resulting materialization digest; mutable Git references, credentials, absolute/traversing paths, unsupported protocols, and executable resolution fields are rejected.
- `schema.ts` owns strict versioned activation records, including the authoritative npm package identity or credential-free Git repository identity required for generic persisted records and Catalog replay, dependency-resolution references, and Catalog curation/source-locator links. `requested_target` is retained as sanitized explanatory provenance rather than source authority, then projected with the record into safe inventory entries.
- `store.ts` deterministically hashes package trees, serializes lifecycle mutations, atomically publishes content-addressed materializations and records, and collects only unreferenced pins.
- `service.ts` makes `GenericExtensionPackageInstaller` the sole owner of direct install/update acquisition-and-publication and Catalog exact installation; it also resolves Catalog-authoring candidates and exact replay provenance. `DirectExtensionService` is an API-compatible thin wrapper for direct install/update plus list/uninstall. Direct candidates are materialized and intrinsically validated before the lifecycle lock; the lock re-reads install/update state (including update target drift), loads active state, validates the complete registry, publishes bytes, and atomically writes the record with its resolution-artifact reference. Exact installs similarly validate locator, curation, immutable-local containment, lock byte length, replay materialization, and selection before their locked conflict/pre-commit-guard/registry/publication step. Staging cleanup is guaranteed. Uninstall guards Sources whose Adapters would disappear.
- `source-bindings.ts` queries only the Source identity and Adapter binding needed by the removal guard through a caller-owned retained database handle; it never opens SQLite itself.

## Data and control flow

`DirectExtensionService.install`/`update` delegate to a short-lived `GenericExtensionPackageInstaller`; `list` reads loadable records and `uninstall` performs the Source-binding guard under the lifecycle lock. Direct install/update materialize a canonical dependency-resolution artifact, derive the authoritative npm or credential-free Git identity (with `requested_target` merely explanatory), select and intrinsically validate the requested root before locking, then re-read lifecycle state and validate/publish atomically through the installer. Catalog authoring returns a selected package root with those exact replay identities and lock bytes for the Catalog manifest. Exact installation validates its locator/provenance, bounds local paths to the immutable snapshot, re-materializes from the canonical lockfile with frozen production dependencies, verifies the stored npm package/version/integrity or Git repository/commit and the materialization digest, and validates the selected root before locking; inside the lock it re-reads records and active roots, rejects origin collisions, validates the combined registry, publishes content-addressed bytes, then writes the generic activation record atomically. Every successful, failed, or pre-publication path cleans staging. Startup derives managed roots from valid records, verifies each digest, and routes them through the common Extension loader without contacting the package manager, network, or original local path.

## Integration points

Exported as `@ctxindex/core/direct-extension`; direct CLI lifecycle handlers supply locked runtime validation context, local OAuth App identities, and Source bindings. Catalog authoring/installation composes `GenericExtensionPackageInstaller` for package resolution and exact replay. `extension/loader.ts` consumes valid records for offline activation.
