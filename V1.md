# ctxindex V1 — milestone scope

Status: draft

This document defines what ctxindex **v1** ships and what it intentionally defers. It does not weaken or replace any requirement in `SPEC.md`. Every behavior here is a restriction on, or selection from, the options the spec allows.

Future milestones (v1.1, v1.x, 2.0, ...) get their own files (`V1_1.md`, `V2.md`, ...).

## 1. Must-ship

### 1.1 Core data plane

- Core SQLite schema with namespaced migrations as defined in `IMPLEMENTATION.md` §3.
- `items`, `chunks`, `external_refs`, `item_relations`, `tombstones` semantics per `SPEC.md` §3–§4.
- `accounts`, `account_identities`, `grants`, `realms` (seeded `global`), `sources`.
- `sync_runs`, `sync_run_checkpoints`, `source_sync_state` per `SPEC.md` §8.
- `sync_locks` with **only `scope = 'global'`** written by v1 (schema accepts `source:<source_id>` for forward compatibility; v1 MUST NOT write per-source rows).
- ULID-only primary keys, client-side generated.

### 1.2 Adapter registry

- The typed `CtxindexAdapterRegistryHandle` from `IMPLEMENTATION.md` §3d.2 is the **only** API for adapter lookups across the codebase.
- v1 registry contents: `local.directory` and `google.mailbox`.
- No runtime `registerAdapter` from the CLI binary in v1; runtime register/unregister is reserved for tests.

### 1.3 Source adapters

- `local.directory`: end-to-end. Config schema, walking via `fdir`, ignore matching via `ignore` with `.gitignore` + `.ctxindexignore`, MIME/binary sniff via `file-type`, SHA-256 hashes, line/block chunking, sync run + checkpoints. Defaults defined in §1.3.1.
- `google.mailbox`: end-to-end. OAuth2 via `google-auth-library`, History API + Zod-validated raw `fetch`, `letterparser` for RFC822, `linkedom` for HTML→text, body + chunks, attachment metadata, RFC822 `Message-ID` external ref, sync run + checkpoints. Defaults defined in §1.3.2.

#### 1.3.1 `local.directory` defaults

Built-in default ignore globs (in addition to user `.gitignore` and `.ctxindexignore`):

```text
.git/
node_modules/
.venv/
.tox/
__pycache__/
dist/
build/
target/
.next/
.nuxt/
.svelte-kit/
.turbo/
.cache/
.parcel-cache/
.DS_Store
Thumbs.db
*.lock
package-lock.json
bun.lock
bun.lockb
pnpm-lock.yaml
yarn.lock
poetry.lock
Cargo.lock
uv.lock
```

Limits:

- Default per-file size cap: **2 MB**. Files larger are skipped and counted in `sync_runs.errors_count` with a `skipped:size` warning.
- Binary detection: `file-type` magic-byte sniff; binary files are skipped unless their MIME is on the text-treatable allowlist (`application/json`, `application/xml`, `application/x-yaml`, `application/toml`, `application/x-sh`, etc.).
- Symlinks: not followed in v1.
- Hidden files (leading `.`): indexed unless ignored by the default set or user rules.

Chunking parameters (shared by all extracted-text sources in v1):

- Target chunk size: **~1500 characters**.
- Overlap: **~200 characters**.
- Preferred break order: paragraph (blank line) > markdown/code heading > newline > hard split.
- Hard split if no break is found within a 3000-character window.

#### 1.3.2 `google.mailbox` defaults

OAuth client credentials:

- v1 is **bring-your-own** (BYO). The user supplies `client_id` and `client_secret` from their own Google Cloud project at `ctxindex auth add google`. ctxindex MUST NOT ship a bundled Google OAuth app in v1.
- A bundled OAuth app may be added in a later milestone after Google verification; deferring it avoids the verification gate while ctxindex is unpublished.

Sync strategy:

1. First sync per source: full backfill via `users.messages.list` with `q` scoped to default label set, paginated. Stores the latest `historyId` seen.
2. Subsequent syncs: incremental via `users.history.list` from the stored `historyId`. On `404` / `historyId too old`, the adapter falls back to a bounded resync (re-list with a `before:`/`after:` window covering the gap) and surfaces a `resync_required` warning op.
3. Default backfill label scope: `INBOX`, `SENT`, `IMPORTANT`, plus any user-labeled threads. Excluded by default: `SPAM`, `TRASH`, `CHAT`, `DRAFT`.
4. Per-source `labels_include` / `labels_exclude` config keys override the defaults. v1 ships the defaults; custom label filters work but are not surfaced in `source add` interactive UX (set via config or `--config-json`).

Attachments:

- Attachment metadata (filename, MIME, size, content-id) is always recorded.
- Attachment body extraction default cap: **25 MB**. Larger attachments record metadata only and increment `sync_runs.errors_count` with a `skipped:size` warning.
- Non-extractable attachment types (executables, encrypted archives, unknown binary) record metadata only.

Raw payloads:

- Off by default per SPEC §9. Enabled per-source via `raw_records_enabled = true` in source config.

### 1.4 Search

- FTS5 over `items_fts` and `chunks_fts` as defined in `IMPLEMENTATION.md` §7.
- BM25 ranking; sanitized query with relaxed-query fallback.
- `--explain` output showing which index paths contributed.
- Filters: source, source adapter, provider module, account, realm, item kind, time range, `--include-deleted`.

### 1.5 Identity & multi-account

- Multiple accounts per provider.
- Multiple grants per account (different scope sets for different sources).
- Multiple sources per account/grant across realms where the provider permits it.
- Mailbox sources classify sent/received via account identities.

### 1.6 CLI surface

Commands shipped in v1:

```text
ctxindex init
ctxindex auth add <provider> [--client-id <id>] [--client-secret <secret>] [--auth-code <code>]
ctxindex auth list [--json]
ctxindex realm add <slug>
ctxindex realm list [--json]
ctxindex source add <adapter-id> --realm <slug> [adapter flags] [--config-json <json>]
ctxindex source list [--realm <slug>] [--json]
ctxindex source remove <source-id>
ctxindex sync [--source <id>] [--mode sync|resync|diff]
ctxindex search <query> [--realm ...] [--source ...] [--adapter ...] [--kind ...] [--since ...] [--until ...] [--include-deleted] [--explain] [--json]
ctxindex status [--source <id>] [--json]
ctxindex secrets migrate <backend>
ctxindex skills list | get <name> [--inline] | path
```

Every read command supports `--json`. Default output is compact one-line text.

**No interactive prompts.** Per SPEC §10b, the v1 CLI MUST NOT use TTY prompts for required input. Concretely:

- `ctxindex source add` requires `--realm <slug>`. If omitted, the source lands in `global`. If `--realm <slug>` references a non-existent realm, the command MUST fail with exit code `2` and an actionable error: `unknown realm "work"; create it with: ctxindex realm add work`.
- `ctxindex auth add google` accepts `--client-id` / `--client-secret` directly (BYO per §1.3.2). The OAuth browser flow MAY open a browser when a TTY is detected; headless callers MUST be able to pass `--auth-code <code>` after completing the authorization step out-of-band. The code-exchange endpoint MUST also be reachable via a loopback redirect URI handled by a short-lived local listener bound to `127.0.0.1:0`.
- `ctxindex secrets migrate <backend>` fails if a passphrase is required for the encrypted-file backend and none is provided via `--passphrase` or `CTXINDEX_SECRETS_PASSPHRASE`.

Partial-success error reporting on sync:

- Exit code is `0` whenever `sync_runs.status = 'completed'`, regardless of `errors_count`.
- The default one-line summary prints `errors_count` when non-zero.
- The full error list is only available via `--json` output or `ctxindex status --source <id> --json`.
- Terminal failures use the exit codes from SPEC §12 (`10` needs-auth, `20` rate-limited, `30` network/provider, `40` permission denied, `50` other, `130` cancelled).

### 1.7 Configuration, secrets, paths

- Single TOML config at `$XDG_CONFIG_HOME/ctxindex/config.toml`.
- Data at `$XDG_DATA_HOME/ctxindex/` (SQLite, raw_records).
- Logs at `$XDG_STATE_HOME/ctxindex/logs/` (rotating JSONL via pino-roll).
- Cache at `$XDG_CACHE_HOME/ctxindex/`.
- `paths` module is the only legal resolver.
- Secrets: keychain backend when available; encrypted `secrets.box` (`@noble/ciphers` xchacha20-poly1305) fallback. Backend chosen at `ctxindex init`; migratable via `ctxindex secrets migrate <to>`.

### 1.8 Observability

- pino + pino-pretty. Levels trace..fatal, default info. `--log-level` / `CTXINDEX_LOG_LEVEL`.
- Stderr (pretty TTY / JSON when piped) + rotating JSONL file.
- Per-record enrichment `{ runId?, sourceId?, adapterId?, accountId?, realmId?, op? }` via adapter-scoped child loggers.
- Pino redaction for `*.access_token`, `*.refresh_token`, `*.authorization`, `*.cookie`, `*.password`, `*.apiKey`.

### 1.9 Bundled skills

- Skills directory shipped with the binary, accessible via `ctxindex skills`.
- Versioned with the v1 release.

## 2. Explicit deferrals

The following are intentionally **out of v1**:

- **Adapters**: `microsoft.mailbox`, `google.calendar`, `microsoft.calendar`, `clickup.tasks`. Next-target set is captured in §3.
- **Vector embeddings and hybrid search**: FTS5 only in v1. `embedding_jobs` / `chunk_embeddings` tables are not created in v1.
- **Cross-source duplicate collapse**: each source's copy is a separate item even when `rfc822_message_id` matches across accounts.
- **Per-source concurrency**: v1 takes only the global sync lock.
- **Rename detection** in `local.directory`: rename surfaces as tombstone + new item.
- **AST / tree-sitter code chunking**: line/block chunking only.
- **Extractors for PDF, DOCX, OCR, audio, video**.
- **`doctor` command, MCP server, daemon, TUI**.
- **`maintenance purge` automation**: manual command only; never auto-runs.
- **Stable export format**: v1 MAY ship `ctxindex export`, but it MUST be marked unstable.
- **Distribution channels beyond `bun link`**: no npm publish, no compiled single binary, no Changesets/bumpp/release automation in v1.

## 4. Exit criteria (v1 "done")

v1 is considered shipped when:

1. `bun link` in a fresh repo gives a working `ctxindex` binary on macOS and Linux.
2. `ctxindex init` creates config, data, state, cache, and secrets directories with correct XDG paths and permissions.
3. `ctxindex auth add google` produces a usable grant; `google.mailbox` sync runs end-to-end against a real Gmail account, producing items + chunks + attachments + sync_run + checkpoint rows.
4. `ctxindex source add local.directory <path>` indexes a real source tree honoring `.gitignore` and `.ctxindexignore`, with file size / binary skip counts reflected in the sync run.
5. `ctxindex search "<query>"` returns ranked items with chunk excerpts and supports every filter listed in §1.6, including `--explain` and `--json`.
6. A crashed sync (SIGKILL during run) leaves `sync_locks` releasable at next startup and does not corrupt `source_sync_state`.
7. Re-auth flow works: revoking the Google token mid-sync produces `last_status = needs_auth` and exit code `10`; `ctxindex auth add google` recovers.
8. Logs land in `$XDG_STATE_HOME/ctxindex/logs/` with rotation, no secrets present after grepping `access_token`/`refresh_token`/`authorization` in rotated files.
9. `ctxindex skills list` / `get` / `path` work from the bundled binary.
10. No network egress to any host that is not a declared provider API for an active source (verified by traffic capture during a sync).
11. `ctxindex source add local.directory <path>` without `--realm` lands the source in `global`; with `--realm unknown` fails with exit code `2`; with `--realm work` (after `ctxindex realm add work`) succeeds. No interactive prompts triggered in any code path.
12. Headless `ctxindex auth add google --client-id ... --client-secret ... --auth-code ...` completes without opening a browser.

Until all twelve hold, v1 is not done.

## 5. Distribution

- v1 ships via `bun link` from a checked-out repo only.
- No npm package, no compiled binary, no installer in v1.
- Release tooling (Changesets, bumpp, GitHub Releases, brew tap, etc.) is selected after v1 once schema, sync_runs contract, and registry have stabilized.
