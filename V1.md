# ctxindex V1 — milestone scope

Status: draft

V1 is the first product version of ctxindex. The current repository code and local databases are disposable prototype scaffolding, not an earlier public version. V1 requires no data migration, compatibility layer, deprecated vocabulary, or preservation of prototype CLI behavior.

## 1. Product outcome

An agent can use one deterministic interface to discover, retrieve, and locally materialize context across user-defined Realms, then persist a reversible email Draft through the same configured mailbox Source and authentication.

## 2. Must ship

### 2.1 Definition and extension runtime

- Public `defineProfile`, `defineAdapter`, and `defineExtension` factories.
- Runtime-validated Profile, Adapter, and Extension registries.
- Explicit-path loading of trusted external TypeScript/JavaScript Extensions.
- Compiled Bun binary support for external TypeScript loading; Bun is pinned to 1.3.14 and the D3 regression check remains green.
- Registry-derived kinds, fields, formats, Source configuration, Actions, and generated documentation; no parallel hand-maintained command definitions.

### 2.2 Realms, Sources, and authentication

- User-defined Realms such as `personal`, `company`, and `university`.
- Every Source belongs to exactly one explicitly selected Realm.
- No seeded or implicit `global` Realm: no realm filter means all Realms, while `--realm company` means company only.
- Persisted provider-scoped OAuth Clients, globally labeled Accounts, one stable internal Grant per Account, and explicit compatible Source-to-Grant binding.
- Globally labeled Sources with verbatim defaults and label-or-id references.
- Declarative auth kinds required by proving adapters, with secret values outside SQLite and typed references persisted locally.

### 2.3 Generic local data plane

- Resources, typed field index, chunks plus FTS, Relations, Artifacts, and Source/Sync bookkeeping.
- No per-domain tables and no adapter-private tables.
- Stable `ctx://` Refs shared by synced and ad-hoc resources.
- Synced and ad-hoc materialization with purgeable local state.
- No migration from the prototype schema; development databases are deleted and recreated.

### 2.4 Uniform access operations

- `sync` for adapters that support maintained local projections.
- local and provider-side `search` through one result envelope and filter grammar.
- `get <ref>` for complete Resource retrieval.
- relation traversal sufficient for `thread get`.
- Artifact download and Profile-declared export.
- token-efficient deterministic JSON and stable error/exit behavior.

### 2.5 Typed provider Actions

- Profiles may declare typed Actions; Adapters implement supported Actions through the same Source/Auth boundary as reads.
- V1 ships only reversible provider-persisted email Draft Actions:
  - `communication.message.draft.create`
  - `communication.message.draft.update`
- Draft Actions require an explicit mailbox Source and return the resulting Resource Ref.
- Text composed only in an agent conversation is not a provider Draft and requires no ctxindex call.
- Sending email and other externally consequential mutations are deferred.

### 2.6 Proving verticals

- `google.mailbox`: message discovery/retrieval, thread relations, attachments/exports, and provider Draft create/update.
- `local.directory`: file sync/search/retrieval through the same Profile/Resource contract.
- One external Extension (initially tenders) loaded outside the compiled binary to prove that bundled and external definitions use the same public seam.

## 3. Target CLI vocabulary

```text
ctxindex init
ctxindex realm add|list|remove
ctxindex client add <provider> [--label <label>] --from-env
ctxindex client list
ctxindex client remove <provider> <label>
ctxindex account add <provider> [--label <label>] [--client <label>]
ctxindex account list [--json]
ctxindex account remove <label>
ctxindex source add <adapter-id> --realm <slug> [--account <label|id>] [--label <label>] [adapter flags]
ctxindex source list|remove
ctxindex sync [--source <label|id>]
ctxindex search [--source <label|id>]
ctxindex get
ctxindex thread get
ctxindex artifact list|download
ctxindex export
ctxindex action describe|run [--source <label|id>]
ctxindex status [--source <label|id>]
ctxindex purge
ctxindex extensions list
ctxindex describe
ctxindex skills list|get|path
ctxindex secrets status
ctxindex secrets backend set
```

Client labels are provider-scoped; Account and Source labels are globally unique bare handles. Source-referencing commands accept labels or ids. Adapter-specific options derive from loaded definitions, and required input is always expressible non-interactively through arguments, declared environment variables, typed secret references, or declared stdin.

## 4. Explicitly deferred

- Sending email.
- Calendar, task, file, or arbitrary provider mutations.
- Arbitrary Extension-registered CLI subcommands.
- Out-of-process or non-TypeScript adapter protocols.
- npm/git Extension distribution and dependency resolution.
- An MCP server transport; the V1 CLI is the canonical agent interface.
- Extension-private storage.
- Multiple primary Profiles on one Resource.
- Cross-source identity deduplication.
- Vector/semantic search, change notifications, and provider write workflows.
- Data migration or compatibility with prototype databases and CLI behavior.

## 5. Implementation slices

Each slice must produce one independently verifiable behavior before the next begins:

1. definition factories and typed registries;
2. explicit-path Extension loading;
3. generic Resource storage with one fake Profile;
4. minimal Gmail message search and get;
5. thread Relations;
6. Artifacts and export;
7. provider Draft create/update Actions;
8. local-directory Adapter;
9. external tenders Extension proof;
10. generated docs and final end-to-end validation.

## 6. Exit criteria

V1 is complete when all must-ship behavior is implemented through public contracts, the compiled binary passes the external-extension regression, fresh-database unit/integration/e2e suites pass, human-assisted live Gmail search/get and Draft create/update checkpoints pass in an isolated sandbox without sending mail, generated CLI/agent docs match the registries, and the active OpenSpec change is verified, synchronized, and left ready for a separate explicit archive request.
