# System Reference Skill

Create and maintain `SYSTEM.md` as the human-readable projection of ctxindex as a whole.

`SYSTEM.md` is agent-authored prose. It is not generated by a script, and this workflow does not produce a PDF or other rendered copy.

## Authority and Source Order

Use these sources according to their ownership:

1. `openspec/specs/<capability>/spec.md` is canonical for normative behavior.
2. `CONTEXT.md` is canonical for ubiquitous language and domain relationships.
3. `openspec/specs/<capability>/implementation.md`, when present, records selective intended-implementation doctrine. Its primary content is real TypeScript interface listings for public surfaces and inter-module seams. It is non-normative and may be absent for a capability.
4. `docs/` explains rationale and usage. `docs/milestones/` contains historical milestone records, not the timeless whole-system contract.
5. `SYSTEM.md` is a readable projection of those sources. It never overrides them.

If `SYSTEM.md` conflicts with a capability spec, fix `SYSTEM.md`; do not edit the spec to match the projection. If two canonical sources genuinely conflict, report the conflict instead of choosing or inventing behavior.

## Required Header

Begin `SYSTEM.md` with a prominent header that states:

- **NON-NORMATIVE**: this document is a readable projection;
- on conflict, `openspec/specs/<capability>/spec.md` wins;
- the last-refreshed date in `YYYY-MM-DD` form;
- the sources consulted for that refresh.

Name source groups precisely. Record which implementation sidecars were present, or state that none were consulted. Do not imply that an absent or changing sidecar was available.

## Required Structure

Write the following sections in this order. Merge subsections only when doing so improves readability without hiding the subject.

1. **Overview and value proposition** — what ctxindex is, who it serves, which systems remain canonical, and what it deliberately is not.
2. **Domain model** — distill the ubiquitous language from `CONTEXT.md`, including the important entity relationships. Use the canonical capitalized terms.
3. **Trust boundaries and security model** — local data boundary, secret storage, approved network egress, logging/redaction, Extension trust, and why Realms are not security boundaries.
4. **Extension architecture** — explain Profiles, Source Adapters, Extensions, registries, capabilities, host-provided contexts, degraded loading, and module ownership.
5. **Accounts, Clients, Grants, and Realms** — explain identities, labels, OAuth selection and reauthorization, Source binding, Realm membership, and removal behavior.
6. **Search and sync behavior** — explain local/federated/hybrid routing, exact filters, ranking and degradation, retrieval/materialization, sync transactions, cursors, locks, and tombstones.
7. **Provider coverage and limitations** — state only provider and local-adapter behavior established by capability specs. Make read-only boundaries and missing dedicated capability contracts visible.
8. **Typed Actions and Drafts** — explain registry-derived Actions, explicit Source selection, validation, effect classes, reversible email Draft scope, and deferred mutations.
9. **Storage model** — explain generic resources, Refs, payloads, fields, chunks, Relations, Artifacts, origins, timestamps, identifiers, retention, backup, and canonical-source boundaries.
10. **CLI surface and stable exit codes** — describe non-interactive behavior, derived command vocabulary, output expectations, entity resolution, bundled skill discovery, and every stable exit code.
11. **Known limitations and deferrals** — gather explicit deferrals, optional future work, unsupported cases, and areas where specs intentionally stay silent. Do not turn optional behavior into a promise.
12. **Source index** — map every preceding `SYSTEM.md` section to the exact capability specs, implementation sidecars, and explanatory docs it distills.

## Initial Authoring Workflow

1. Inventory `openspec/specs/*/spec.md` and confirm every capability is represented.
2. Read `CONTEXT.md` fully.
3. Read every capability `spec.md` fully, including scenarios. Requirements state the contract; scenarios often define important edge behavior.
4. Read every `implementation.md` sidecar that exists at that moment fully. Treat interface listings as intended seams, never as stronger than behavior specs, and do not require a sidecar to exist.
5. Read only the explanatory or historical documents needed to clarify rationale, current scope, or an explicit limitation. Label historical milestone evidence as historical.
6. Build a section-to-source matrix before drafting. Every material claim must be traceable to at least one consulted source.
7. Write for a technical human who wants a coherent mental model. Prefer direct prose, short examples, tables, and explicit boundaries over requirement-by-requirement paraphrase.
8. Preserve normative strength without writing in spec-speak: distinguish required behavior from recommendations, optional facilities, and deferred work.
9. Add the source index and verify that every capability spec appears at least once or is explicitly identified as not contributing to the current projection.
10. Re-read the draft against the sources. Remove invented behavior, accidental guarantees, provider assumptions, and stale milestone claims.

## Refresh Workflow

A refresh is not a search-and-replace date bump.

1. Inspect the changed capability specs and any changed sidecars or explanatory docs.
2. Re-read each changed source fully, plus adjacent capability specs needed to understand cross-links.
3. Rewrite every affected section for coherence; do not append a patch note or duplicate old and new behavior.
4. Check cross-cutting sections such as security, provider coverage, CLI exits, and limitations even when the changed source appears narrow.
5. Update the source index if ownership or consulted files changed.
6. Update the last-refreshed date only after the projection has been checked.
7. Run `bunx openspec validate --all --strict`. This validates OpenSpec artifacts, not the truth of `SYSTEM.md`, so also perform the manual audit below.

## Audit Checklist

Before finishing, confirm:

- the header says **NON-NORMATIVE**, gives a date, names consulted sources, and points conflicts to capability specs;
- all capability specs were read and covered;
- terminology matches `CONTEXT.md` (`Resource`, not prototype `Item`; `Extension`, not plugin; no `global` Realm);
- Profile semantics and Adapter I/O are not assigned to core or Extension composition roots;
- every Source belongs to exactly one Realm, while Realm is not presented as a credential boundary;
- external services and files remain canonical, with local materializations described as purgeable where specified;
- V1 mutations stop at provider-persisted email Draft create/update and no text-in-chat is called a Draft;
- provider failures do not incorrectly imply all search fails when local results survive;
- optional or recommended behavior is not described as shipped or guaranteed;
- limitations are candid, especially where no dedicated capability spec exists;
- the source index covers every section with exact paths;
- no requirement appears only because it seemed desirable.

Tiny inventory or coverage scripts may be added under this skill's `scripts/` directory when useful, but they may only list or compare sources. They must never generate `SYSTEM.md` prose.
