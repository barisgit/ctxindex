# ctxindex V1.1 — multi-provider context scope

Status: Completed — V1.1 shipped on 2026-07-16.

Historical record: This document describes the milestone as planned, not current system truth. Current normative behavior lives in `openspec/specs/`.

V1.1 extends the completed V1 context-access layer. V1 behavior, generic contracts, exit meanings, and no-send boundary remain valid. The repository is still pre-release: prototype databases and CLI aliases are disposable, and this milestone adds no compatibility migration.

## 1. Product outcome

An agent can inspect and use multiple Google and Microsoft Accounts across explicit Realms through one deterministic interface: personal/work Gmail and Outlook mailboxes, Google and Microsoft calendars, local files, and reversible Gmail/Outlook Drafts. Authorization requests provider base scopes plus every loaded same-provider Source Adapter's permissions, secret storage never changes implicitly, and ctxindex cannot send mail or mutate calendars.

## 2. Must ship

### 2.1 Secret storage and authorization foundation

- Explicit Keychain/encrypted-file backend status and crash-safe switching.
- No silent backend fallback and no long-lived secret values in process arguments.
- Provider-neutral declarative OAuth with persisted labeled Clients, one-time credential import from declared environment names, state validation, PKCE where declared, refresh-token rotation, stable provider identity, and bounded hosts.
- Exact sorted scope union from provider base scopes plus every loaded same-provider Adapter; other providers' Adapters contribute no permission.
- One globally labeled Account per stable `(provider, external identity)`, optional verified Account Identities, one stable Grant updated in place, and compatible Source-to-Grant binding.
- Globally labeled Sources with verbatim defaults and deterministic `account list` inventory nesting safe Grant and Source/Realm metadata without secret references or values.

### 2.2 Calendar context

- Canonical strict `calendar.event@1` Profile for timed and all-day events, attendees/organizer, status, recurrence/series linkage, typed fields, chunks, Relations, aliases, and documentation.
- One calendar Source selects one provider calendar and belongs to one explicit Realm.
- `google.calendar`: read-only indexed synchronization and retrieval using stable event Refs, paging, incremental tokens, tombstones, and bounded reconciliation after token invalidation.
- `microsoft.calendar`: read-only indexed synchronization and retrieval using stable Graph v1.0 calendar-view delta over an explicit rolling window.
- No calendar Action, write scope, or mutation endpoint.

### 2.3 Microsoft Graph mailbox

- Microsoft identity authorization for app registrations supporting approved Outlook.com personal and Microsoft 365 organizational Accounts.
- `microsoft.mailbox`: provider-side message search, complete retrieval, conversation Relations, attachment descriptors/download/cache, EML/JSON export, and immutable Source-scoped Refs.
- Exact use of Graph immutable ids on relevant message, Draft, attachment, event, search, paging, and delta requests.
- Existing provider-neutral `communication.message.draft.create` and `.draft.update` Actions implemented for Outlook with one non-retried mutation each and a stable Draft Ref.
- Delegated `Mail.ReadWrite` for Draft persistence, never `Mail.Send`; no send/reply-send/forward-send binding or route.

### 2.4 Agent-first workflows

- `client add/list/remove` manages persisted provider-scoped OAuth Clients; `account add/list/remove` manages globally labeled provider identities while Grants remain internal.
- `source add/list/remove` uses globally unique labels and provider-matched Account label/ID/Grant-ID resolution; `source list` remains the complete configured Source inventory including unauthenticated Sources.
- Generated `describe`, help, skills, and workflow guidance expose loaded Profile/Adapter/auth/config/Action truth without parallel provider command vocabularies.
- A relocated compiled CLI proves personal Gmail/Google Calendar, work Gmail/Outlook/Microsoft Calendar, and local-directory Sources across exact Realms through generic sync/search/get/thread/artifact/export/Action commands.

## 3. Target CLI additions and changes

```text
ctxindex secrets status [--json]
ctxindex secrets backend set <keychain|file>
ctxindex client add <provider> [--label <label>] --from-env
ctxindex client list
ctxindex client remove <provider> <label>
ctxindex account add <provider> [--label <label>] [--client <label>]
ctxindex account list [--json]
ctxindex account remove <label>
ctxindex source add <adapter-id> --realm <slug> [--account <label|id>] [--label <label>] [--no-sync] [adapter flags]
ctxindex source list|remove
```

V1.1 excludes `secrets migrate`, literal secret/token/code arguments, and the fused `auth` command. Client labels are unique per provider; Account and Source labels are globally unique verbatim handles with hard exit-2 collisions. Source references accept labels or ids, while Adapter configuration continues to derive from loaded schemas.

## 4. Explicitly deferred

- Sending mail or requesting `Mail.Send`/provider send-only permissions.
- Calendar create/update/delete/RSVP/invite/notification Actions.
- On-premises Exchange, IMAP/SMTP, service-account/application permissions, webhooks, and push subscriptions.
- Google/Microsoft Drive, OneDrive, contacts, tasks, calendar-list discovery commands, or account-wide aggregate Sources.
- Cross-source Resource deduplication and recurring-rule expansion beyond provider-emitted occurrences/series metadata.
- Microsoft beta whole-calendar delta or beta calendar-list delta.
- Data migration or deprecated CLI aliases for prototype state.

## 5. Implementation slices

Each Slice gate is mandatory before the next dependent Slice:

1. V1.1 contracts and architecture/security guards;
2. explicit Secret Vault/backend operations;
3. provider-neutral OAuth, Account, and Grant core;
4. agent-first client/account CLI;
5. `calendar.event@1` Profile;
6. Google Calendar mocked gate and Human read checkpoint;
7. Microsoft identity and mailbox reads;
8. Outlook Draft Actions;
9. Microsoft Calendar;
10. cross-provider compiled workflow, Microsoft Human read/Draft checkpoint, and final verification.

At each Human checkpoint the agent prepares isolated ignored state and exact scopes/requests, pauses before login/consent or provider mutation, never requests credentials, resumes only after explicit approval, preserves redacted evidence, and waits for provider-UI confirmation where applicable.

## 6. Exit criteria

V1.1 is complete when all must-ship behavior is implemented through public registry/Profile/Adapter/core contracts; exact-scope Google and Microsoft authorization works in mocks; isolated Human checkpoints prove harmless live Google calendar read plus Microsoft mailbox/calendar reads and one Outlook Draft create/update with user confirmation that nothing was sent; the compiled multi-Account/Realm workflow and provider egress/no-send gates pass; fresh unit/integration/e2e, frozen build, relocated binary, external Extension D3, typecheck, lint, architecture/dependency, drift/cartography, and strict OpenSpec gates pass; generated interfaces match registries; independent review and fresh-context QA approve the settled snapshot; and the active change is verified/synchronized but left unarchived for separate explicit approval.
