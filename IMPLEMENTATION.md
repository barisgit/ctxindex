# ctxindex Implementation Notes

Status: draft

This document describes the intended reference implementation for `SPEC.md`. It is allowed to change as the codebase evolves, but changes must preserve the normative behavior in `SPEC.md` unless the spec changes too.

## 1. Runtime choice

Locked: **Bun + TypeScript** for v1. Node is not a build target.

Rationale:

- single binary via `bun build --compile`, no `node_modules` ship requirement;
- native TypeScript execution, no tsx/tsup in the hot path;
- `bun:sqlite` is bundled, fast, and supports loadable extensions (FTS5 ships built-in; sqlite-vec / sqlite-vss can be loaded later);
- Bun's built-in test runner removes Vitest/Jest from the dep tree;
- the source adapter registry and operation contract benefit from TypeScript literal inference;
- OAuth/API provider clients are faster to assemble in TypeScript than in Rust at this stage.

Rust can be reconsidered later for a packaged binary, high-performance extractors, or native embedding/vector components. It is not the v1 application runtime.

## 2. Storage stack

Locked v1:

- **SQLite** as the local database, opened in WAL mode, `foreign_keys = ON`, `synchronous = NORMAL`, `busy_timeout` configured.
- **`bun:sqlite`** as the driver. No `better-sqlite3`, no `node:sqlite`.
- **Drizzle ORM** via `drizzle-orm/bun-sqlite` as the schema and typed query layer.
- **Drizzle Kit** as the schema diff / migration file generator.
- FTS5 virtual tables, triggers, and future vector indexes are written as **hand-authored SQL migrations** alongside Drizzle-generated ones. Drizzle Kit handles ordinary tables; raw SQL handles FTS5 / virtual / extension-backed objects.

Connection sketch:

```ts
import { Database } from 'bun:sqlite';
import { drizzle } from 'drizzle-orm/bun-sqlite';

const sqlite = new Database(dbPath, { create: true });
sqlite.exec('PRAGMA journal_mode = WAL;');
sqlite.exec('PRAGMA foreign_keys = ON;');
sqlite.exec('PRAGMA synchronous = NORMAL;');
sqlite.exec('PRAGMA busy_timeout = 5000;');

export const db = drizzle(sqlite, { schema });
```

Drizzle is used everywhere normal table access is needed (inserts, updates, selects, repository functions are type checked). FTS5 / triggers / virtual tables are accessed through narrow typed wrappers around raw SQL. No Prisma. No MikroORM. No TypeORM.

## 3. Migration architecture

Locked v1: **Drizzle's `migrate()` runner driven by a thin ctxindex coordinator**, one Drizzle call per namespace. No hand-rolled migration engine. No bespoke `schema_migrations` table — each namespace gets its own Drizzle journal table.

### 3.1 Namespaces

Each migration namespace is one logical owner of tables. Examples:

```text
core                 # core identity, sync_runs, realms, accounts, items, FTS
google.mailbox       # mail_* and google_mail_* tables
local.directory      # local_directory_* tables
```

Namespaces match the adapter id (`<provider>.<adapter>`) for adapters, and the literal `core` for ctxindex core. Names are stable for the life of the database; renaming a namespace requires a manual data migration.

### 3.2 Per-namespace migration folders

Each namespace owns a folder of SQL migrations generated by Drizzle Kit, plus optional hand-authored SQL files for FTS5/virtual/vector objects in the same folder. Layout:

```text
packages/core/migrations/
  0000_init.sql
  0001_items.sql
  0002_items_fts.sql                    # hand-authored FTS5 virtual + triggers
  meta/_journal.json                    # Drizzle journal

packages/adapter-google-mailbox/migrations/
  0000_mail_messages.sql
  0001_mail_bodies_attachments.sql
  meta/_journal.json

packages/adapter-local-directory/migrations/
  0000_local_directory_file_state.sql
  meta/_journal.json
```

Drizzle is configured per namespace via separate `drizzle.config.ts` files. Hand-authored SQL files are picked up by Drizzle Kit's `custom` migration support when they live in the namespace's migrations folder.

### 3.3 Per-namespace journal tables

Drizzle `migrate()` accepts `migrationsTable` per call. Each namespace uses its own:

```text
ctxindex_migrations_core
ctxindex_migrations_google_mailbox
ctxindex_migrations_local_directory
```

Adapters can be added, upgraded, or removed without disturbing core's journal, and core upgrades don't touch adapter journals. No shared `schema_migrations` table.

### 3.4 Adapter migration registration

Each adapter exports its migration metadata as part of its registration record. The coordinator iterates registered adapters and applies their migrations in registration order, after core.

```ts
export interface AdapterMigrations {
  namespace: string;                    // e.g. 'google.mailbox'
  migrationsFolder: string;             // absolute path resolved by bundle
  migrationsTable: string;              // e.g. 'ctxindex_migrations_google_mailbox'
}

export interface SourceAdapterModule {
  id: string;                           // 'google.mailbox'
  migrations: AdapterMigrations;
  schema: Record<string, unknown>;      // Drizzle table exports
  // ...sync(), capabilities(), etc.
}
```

The coordinator (`src/storage/migrator.ts`) applies migrations in deterministic order: core first, then every registered adapter sorted by adapter id. Order between adapters does not matter as long as adapters do not reference each other's tables (forbidden by the boundary rules).

### 3.5 Adapter table prefixing rules (enforced by convention)

Adapter tables MUST be prefixed with the adapter's short tag so the underlying SQLite namespace is collision-free:

```text
google.mailbox      -> google_mail_*    (existing canon names like mail_messages stay only for shared mail schema in core; adapter-private tables get the prefix)
local.directory     -> local_directory_*
microsoft.calendar  -> ms_calendar_*
```

Shared mail / calendar / FTS tables live in core (`mail_messages`, `calendar_events`, `items_fts`, `chunks_fts`, etc.). Adapter-private tables live under the adapter's namespace prefix.

Adapter migrations MUST NOT alter core tables. Core migrations MUST NOT depend on adapter tables. Cross-namespace references are limited to `item_id` and `source_id` foreign keys, both owned by core.

### 3.6 Hand-authored SQL: when and where

Use hand-authored SQL migrations for:

- FTS5 virtual tables and their `INSERT/UPDATE/DELETE` triggers;
- `CREATE TRIGGER` statements Drizzle Kit cannot express idiomatically;
- loadable-extension setup (sqlite-vec / sqlite-vss future);
- data backfills that aren't expressible as Drizzle schema diffs.

Use Drizzle-generated SQL for everything else. The journal is shared between generated and custom files, so order is by filename.

```ts
// packages/core/src/storage/migrator.ts
import { migrate } from 'drizzle-orm/bun-sqlite/migrator';
import { db } from './db';
import { coreMigrations } from '../migrations';
import { CTXINDEX_ADAPTER_REGISTRY } from '@ctxindex/adapters';

export function runMigrations(): void {
  // 1. core first
  migrate(db, {
    migrationsFolder: coreMigrations.migrationsFolder,
    migrationsTable: coreMigrations.migrationsTable,
  });

  // 2. adapters in deterministic id order
  const adapters = [...CTXINDEX_ADAPTER_REGISTRY.list()].sort(
    (a, b) => a.id.localeCompare(b.id),
  );
  for (const adapter of adapters) {
    migrate(db, {
      migrationsFolder: adapter.migrations.migrationsFolder,
      migrationsTable: adapter.migrations.migrationsTable,
    });
  }
}
```

## 3a. CLI framework

Locked: **citty** (UnJS) for command parsing, sub-commands, and `--help` rendering.

Reasons: small, typed, sub-command tree fits `ctxindex auth ...`, `ctxindex source ...`, `ctxindex skills ...` cleanly, integrates with Bun without extra build steps. No commander, no clipanion, no oclif.

## 3b. Provider, parsing, and secrets dependencies

Locked v1 third-party dependencies:

- **HTML → text** (mail bodies, future HTML files): `linkedom`. Used through a thin `htmlToText(html: string): string` helper so we can swap implementations without touching adapters.
- **RFC822 MIME parsing** (when raw records are re-extracted, future IMAP): `letterparser`. Pure ESM, no native deps, handles parts and attachments.
- **Recurrence rules**: `rrule`. RFC5545-compliant; used only for runtime expansion within a bounded window, never to materialize occurrences.
- **Google OAuth**: `google-auth-library`. Used as the `OAuth2Client` only — token refresh, ID-token verification.
- **Google API calls**: raw `fetch` against `https://gmail.googleapis.com/...` with `Authorization: Bearer <token>`. **No `googleapis` package.** Every external HTTP response MUST be parsed through a Zod schema before crossing into ctxindex types. Unknown fields are preserved via `.passthrough()` so they remain available in `raw_records`.

```ts
// packages/adapter-google-mailbox/src/api.ts
import { z } from 'zod';
import type { OAuth2Client } from 'google-auth-library';

const GmailMessage = z.object({
  id: z.string(),
  threadId: z.string(),
  historyId: z.string(),
  internalDate: z.string(),
  labelIds: z.array(z.string()).default([]),
  payload: z
    .object({
      headers: z.array(z.object({ name: z.string(), value: z.string() })),
      parts: z.array(z.any()).optional(),
      body: z
        .object({
          data: z.string().optional(),
          attachmentId: z.string().optional(),
          size: z.number().optional(),
        })
        .optional(),
      mimeType: z.string().optional(),
    })
    .optional(),
}).passthrough();
export type GmailMessage = z.infer<typeof GmailMessage>;

export async function safeFetch<T extends z.ZodTypeAny>(
  auth: OAuth2Client,
  schema: T,
  url: string,
): Promise<z.infer<T>> {
  const { token } = await auth.getAccessToken();
  const res = await fetch(url, { headers: { authorization: `Bearer ${token}` } });
  if (!res.ok) throw new Error(`gmail ${res.status}: ${await res.text()}`);
  return schema.parse(await res.json()) as z.infer<T>;
}
```

The same `safeFetch` pattern is reused for Microsoft Graph and ClickUp later.

### 3b.1 Secrets at rest

Locked v1: **hybrid secrets store**. Order of preference at runtime:

1. **OS keychain via `keytar`** when available (macOS Keychain, GNOME libsecret on Linux, Windows Credential Manager). Best UX on a developer workstation.
2. **Encrypted local file** at `$CTXINDEX_DATA_HOME/secrets.box` (default `~/.local/share/ctxindex/secrets.box`), encrypted with `@noble/ciphers` (xchacha20-poly1305) using a key derived from a passphrase or a device-local key file at `$CTXINDEX_CONFIG_HOME/secret.key` (mode 0600). Paths resolved through the `paths` module, not hard-coded. Used as fallback when keytar is unavailable, when the user opts into a portable single-binary build, or when running headless.

Database and TOML never store cleartext secrets. They store **secret references** of the shape `keychain:ctxindex/<scope>/<key>` or `file:secrets.box#<key>`. The secrets store resolves the reference at runtime.

Selection rules:

- `ctxindex init` probes for keytar; if present and the keychain service is reachable, mark backend `keychain` in the config.
- If keytar is unavailable, prompt the user once: passphrase or device-key file. Store backend = `file`.
- Backend choice is per-installation, not per-secret. Backend can be migrated via `ctxindex secrets migrate <to>`.

Single-binary distribution implications: keytar is a native module and may not be bundleable with `bun build --compile` on every platform. When that path matters, the binary build ships with `keychain` backend disabled at compile time and falls back to the encrypted file. Documented in the release notes.

## 3c. Observability, testing, lint, typecheck

Locked v1:

- **Logger**: `pino` (structured JSON) + `pino-pretty` for the `--pretty` dev flag.
  - In compiled single-binary mode logging uses `pino.destination({ sync: true })` only — no transport workers — to avoid Bun-compile worker-thread edge cases.
  - In `bun run` mode async transports are allowed.
  - Log levels: `trace | debug | info | warn | error | fatal`. Default `info`. Set via `--log-level`, `CTXINDEX_LOG_LEVEL`, or `[log] level` in `config.toml`. CLI `--verbose` raises to `debug`.
  - Two sinks always: human-formatted line to **stderr** (pretty when TTY, JSON when piped or `--json` in effect), structured JSONL to a rotating file under `logDir()` (see section 11).
- **Log rotation**: `pino-roll` with daily rotation, gzip on rotate, retention 14 days by default, configurable via `[log] file = { rotate = "daily", retain_days = 14, compress = true }`. Rotation key included in filename: `ctxindex-YYYY-MM-DD.log.gz`.
- **Log enrichment**: every record carries `{ runId?, sourceId?, adapterId?, accountId?, realmId?, op? }` so a sync trace is reconstructable. Adapter code receives a scoped child logger pre-bound to its `sourceId`.
- **Sensitive fields**: pino redaction list configured for `*.access_token`, `*.refresh_token`, `*.authorization`, `*.cookie`, `*.password`, `*.apiKey`. Redacted at the logger boundary, not at call sites.

- **Tests**: Bun's built-in test runner (`bun test`). No Vitest, no Jest. Tests live next to source as `*.test.ts`. Integration tests for adapters spin a temp SQLite DB. Three lanes via `bunfig.toml` `pathIgnorePatterns` + scripts: `test` (unit), `test:integration` (under `**/*.integration.test.ts`), `test:e2e` (spawns the linked CLI binary). The e2e lane uses a `scripts/with-timeout.ts` wall-clock guard (kills process group, exits `124` on timeout) so a hung Google API call cannot freeze CI. JUnit reporter via `bun test --reporter=junit --reporter-outfile=.cache/bun-junit.xml` is optional in CI.
- **Test harness** (`packages/core/src/testing/`): a `createSandbox()` helper returns `{ envOverrides, paths, openDb, cleanup }` where `envOverrides` is the `CTXINDEX_*` map a child process inherits, `paths` is a resolved `paths` module bound to a tmpdir, `openDb()` opens (or rehydrates from a migrated template) the sandboxed SQLite file, and `cleanup()` removes the tmpdir. Every integration and e2e test MUST use `createSandbox()` rather than ad-hoc `mkdtempSync` calls.
- **Lint + format**: **Biome** (`biome.json` checked in). Single tool, no ESLint/Prettier. CI runs `biome check`.
- **Typecheck**: `@typescript/native-preview` (`tsgo`) is the CI typecheck binary (`tsgo --noEmit`). `typescript` stays as a dev dep so editors/LSP keep working and to retain a `tsc --noEmit` fallback. `tsconfig.json` flags: `strict: true`, `noUncheckedIndexedAccess: true`, `exactOptionalPropertyTypes: true`, `noImplicitOverride: true`, `noFallthroughCasesInSwitch: true`, `verbatimModuleSyntax: true`.
- **Env loader**: a single Zod schema in `packages/core/src/config/env.ts` parses `process.env` into a frozen object via a `getEnv()` singleton; `resetEnvForTests()` clears the cache. Direct `process.env.CTXINDEX_*` reads outside this module are a lint-enforceable v1 violation. The schema covers every `CTXINDEX_*` key plus `XDG_*` and optional Google OAuth fallbacks (`CTXINDEX_GMAIL_CLIENT_ID`, `CTXINDEX_GMAIL_CLIENT_SECRET`) used by the e2e harness.
- **CI sequence**: `bun install` → `biome check` → `tsgo --noEmit` → `bun test` → `bun test:integration` → `bun test:e2e` (gated on presence of CI env vars; falls back to mocked harness when no Google credentials are configured).

## 3d. Repo layout, adapter registry, distribution

### 3d.1 Monorepo shape

Locked v1: **Turborepo monorepo** with `packages/*` for libraries and `apps/*` for runnables.

```text
ctxindex/
  apps/
    cli/                                  # `ctxindex` binary entrypoint, citty commands
  packages/
    core/                                 # storage, schema, identity, sync_runs,
                                          # realms, accounts, search, FTS, paths,
                                          # logger, secrets, registry-core
    adapters/                             # ALL bundled source adapters in one package
      src/local-directory/
      src/google-mailbox/
      src/index.ts                        # registers and exports the built-in registry
      migrations/
        local-directory/
        google-mailbox/
  skills/                                 # bundled skill docs, copied into binary
  turbo.json
  package.json                            # bun workspaces
  biome.json
  tsconfig.base.json
```

Adapters do NOT each get their own package. One `@ctxindex/adapters` package owns every bundled source adapter. Reasons:

- avoids package explosion as the count grows;
- a single Drizzle/Biome/tsconfig setup;
- single npm publish unit;
- per-adapter `migrationsFolder` still scopes migrations cleanly.

Workspace tool: **Turborepo + Bun workspaces** (Turborepo task graph + caching, Bun for installs and runtime). Turborepo pnpm support is more polished today but Bun workspace support is improving and matches our Bun-only stance.

`turbo.json` pipelines: `build`, `lint` (biome), `typecheck` (tsgo), `test` (bun test, unit), `test:integration`, `test:e2e`, `db:generate` (drizzle-kit per namespace).

Root `turbo.json` MUST set `globalDependencies: ['**/.env', '**/.env.*', '**/bunfig.toml']` so any env/config change busts caches. Each workspace (`apps/cli`, `packages/core`, `packages/adapters`) MAY extend `//` and declare an `env:` array per task listing the `CTXINDEX_*` variables that should bust that task's cache; the e2e lane in `apps/cli/turbo.json` MUST list `CTXINDEX_GMAIL_CLIENT_ID`, `CTXINDEX_GMAIL_CLIENT_SECRET`, `CTXINDEX_GMAIL_REFRESH_TOKEN` so live-credential changes do not silently reuse a stale pass.

### 3d.2 Adapter registry

Locked: a **fully-typed, capability-rich adapter registry** using a `createIntegrationRegistry`-style pattern. The registry is constructed once over a `const` map of adapter definitions; from that point on, every capability a consumer might need is a typed method on the returned handle. Consumers do not reach into adapter objects directly. Unknown ids fail at compile time.

Design rules:

1. Adapter ids are TypeScript literal unions inferred from the map (`'local.directory' | 'google.mailbox' | ...`).
2. The handle interface (`CtxindexAdapterRegistryHandle<TAdapters>`) is the **only** API surface for adapter-related lookups across the codebase. No `Object.values(adapters)` or string lookups outside the registry-core module.
3. The construction step pre-builds every index (by id, by provider, by capability, by auth kind, by namespace). No O(n) scans at call time.
4. Every accessor that takes an id takes the **literal-typed id** (`Extract<keyof TAdapters, string>`), not raw `string`.
5. Lookups by free `string` (e.g. resolving a CLI-entered adapter name) go through `isKnownAdapter(v)` first; that's a type guard that narrows `string` to the literal union for subsequent calls.
6. Migration coordinator, sync runner, auth flow, CLI `source add`, and `ctxindex status` all consume the registry through dedicated typed methods (`listMigrations()`, `getSyncFn(id)`, `getAuthSpec(id)`, `getAdaptersByProvider(provider)`, etc.). Each consumer's needs become methods on the handle, not patterns repeated in callers.
7. The handle also exposes typed `registerAdapter` / `unregisterAdapter` runtime hooks (for tests, future out-of-tree adapters). v1 binary ships the built-in map only; runtime hooks are present but unused.
8. All registry errors throw `CtxindexRegistryError` with a `code` + structured `metadata`.

#### 3d.2.1 Adapter definition shape (`packages/core/src/registry/types.ts`)

```ts
import type { z } from 'zod';
import type { Logger } from '@ctxindex/core/logger';

export type AdapterProvider = 'google' | 'microsoft' | 'clickup' | 'local';
export type SyncMode = 'sync' | 'resync' | 'diff';
export type SourceKind = 'mailbox' | 'calendar' | 'directory' | 'tasks';

export interface AdapterMigrations {
  readonly namespace: string;                     // e.g. 'google.mailbox'
  readonly migrationsFolder: string;              // absolute, resolved at module load
  readonly migrationsTable: string;               // e.g. 'ctxindex_migrations_google_mailbox'
}

export interface AdapterCapabilities {
  readonly kinds: readonly [SourceKind, ...SourceKind[]];
  readonly modes: readonly [SyncMode, ...SyncMode[]];
  readonly supportsResume: boolean;
  readonly supportsAttachments: boolean;
  readonly supportsRawRecords: boolean;
  readonly supportsRealm: true;                   // every v1 source is realm-bound
}

export type AdapterAuthSpec =
  | { readonly kind: 'none' }
  | {
      readonly kind: 'oauth2';
      readonly provider: 'google' | 'microsoft';
      readonly scopes: readonly string[];

      readonly clientSecretRef?: string;          // optional for PKCE
    };

export interface SyncContext {
  readonly sourceId: string;
  readonly runId: string;
  readonly mode: SyncMode;
  readonly cursor: unknown | null;
  readonly logger: Logger;
  readonly signal: AbortSignal;
}

export type SyncFunction = (
  ctx: SyncContext,
) => AsyncIterable<SyncOperation>;

export interface SourceAdapterDefinition<TId extends string = string> {
  readonly id: TId;                               // literal in concrete defs
  readonly provider: AdapterProvider;
  readonly label: string;
  readonly schema: Readonly<Record<string, unknown>>; // Drizzle table exports
  readonly capabilities: AdapterCapabilities;
  readonly migrations: AdapterMigrations;
  readonly auth: AdapterAuthSpec;
  readonly sync: SyncFunction;
  readonly configSchema: z.ZodTypeAny;            // per-source-instance config schema
}
```

#### 3d.2.2 Registry handle interface

```ts
export interface CtxindexAdapterRegistryHandle<
  TAdapters extends Record<string, SourceAdapterDefinition>,
> {
  // ---- catalog -----------------------------------------------------------
  readonly adapters: TAdapters;
  readonly adapterIds: readonly [
    Extract<keyof TAdapters, string>,
    ...Extract<keyof TAdapters, string>[],
  ];
  readonly providers: readonly AdapterProvider[];
  readonly namespaces: readonly string[];

  // ---- identity / lookup -------------------------------------------------
  isKnownAdapter(v: string): v is Extract<keyof TAdapters, string>;
  assertKnownAdapter(v: string): asserts v is Extract<keyof TAdapters, string>;
  getAdapter<TId extends Extract<keyof TAdapters, string>>(id: TId): TAdapters[TId];
  getLabel(id: Extract<keyof TAdapters, string>): string;
  getProvider(id: Extract<keyof TAdapters, string>): AdapterProvider;

  // ---- listing / grouping ------------------------------------------------
  listAdapters(): readonly SourceAdapterDefinition[];
  listAdapterIds(): readonly Extract<keyof TAdapters, string>[];
  getAdaptersByProvider(provider: AdapterProvider): readonly SourceAdapterDefinition[];
  getAdaptersByKind(kind: SourceKind): readonly SourceAdapterDefinition[];

  // ---- migrations (used by storage/migrator.ts) --------------------------
  listMigrations(): readonly AdapterMigrations[];
  getMigrations(id: Extract<keyof TAdapters, string>): AdapterMigrations;
  getNamespaceForId(id: Extract<keyof TAdapters, string>): string;
  getAdapterForNamespace(namespace: string): SourceAdapterDefinition | undefined;

  // ---- capabilities ------------------------------------------------------
  getCapabilities(id: Extract<keyof TAdapters, string>): AdapterCapabilities;
  getSupportedModes(id: Extract<keyof TAdapters, string>): readonly SyncMode[];
  supportsMode(id: Extract<keyof TAdapters, string>, mode: SyncMode): boolean;
  supportsResume(id: Extract<keyof TAdapters, string>): boolean;
  supportsAttachments(id: Extract<keyof TAdapters, string>): boolean;
  supportsRawRecords(id: Extract<keyof TAdapters, string>): boolean;

  // ---- sync execution ----------------------------------------------------
  getSyncFn(id: Extract<keyof TAdapters, string>): SyncFunction;
  getSchema(id: Extract<keyof TAdapters, string>): Readonly<Record<string, unknown>>;
  getConfigSchema(id: Extract<keyof TAdapters, string>): z.ZodTypeAny;

  // ---- auth --------------------------------------------------------------
  getAuthSpec(id: Extract<keyof TAdapters, string>): AdapterAuthSpec;
  isOAuth2(id: Extract<keyof TAdapters, string>): boolean;
  listOAuth2Adapters(): readonly SourceAdapterDefinition[];
  listOAuth2AdaptersByProvider(
    provider: 'google' | 'microsoft',
  ): readonly SourceAdapterDefinition[];
  getRequiredScopes(
    id: Extract<keyof TAdapters, string>,
  ): readonly string[] | null;

  // ---- runtime extension (tests + future ecosystem) ----------------------

  unregisterAdapter(id: string): SourceAdapterDefinition | undefined;
```

#### 3d.2.3 Registry core (`packages/core/src/registry/registry-core.ts`)

Constructs every index up front; methods are O(1) lookups thereafter. Errors are typed `CtxindexRegistryError` with a `code` and `metadata`. Sketch (essentials only; full version lives in code):

```ts
import { CtxindexError } from '@ctxindex/core/errors';

type CtxindexRegistryErrorCode =
  | 'registry_unknown_adapter'
  | 'registry_unknown_namespace'
  | 'registry_unknown_provider'
  | 'registry_unsupported_mode'
  | 'registry_mode_required';

export class CtxindexRegistryError extends CtxindexError {
  readonly metadata: Record<string, unknown>;
  constructor(message: string, code: CtxindexRegistryErrorCode, metadata: Record<string, unknown>) {
    super(message, code);
    this.name = 'CtxindexRegistryError';
    this.metadata = metadata;
  }
}

export function createSourceAdapter<
  TId extends string,
  TDef extends Omit<SourceAdapterDefinition, 'id'>,
>(id: TId, def: TDef): TDef & { readonly id: TId } {
  return Object.freeze({ id, ...def }) as TDef & { readonly id: TId };
}

export function createCtxindexAdapterRegistry<
  const TAdapters extends Record<string, SourceAdapterDefinition>,
>(adapters: TAdapters): CtxindexAdapterRegistryHandle<TAdapters> {
  type AdapterId = Extract<keyof TAdapters, string>;

  const adapterIds = Object.freeze(
    Object.keys(adapters) as [AdapterId, ...AdapterId[]],
  );
  const entries = Object.values(adapters) as SourceAdapterDefinition[];

  // pre-built indexes ----------------------------------------------------
  const byNamespace = new Map<string, SourceAdapterDefinition>();
  const byProvider = new Map<AdapterProvider, SourceAdapterDefinition[]>();
  const byKind = new Map<SourceKind, SourceAdapterDefinition[]>();
  const oauth2: SourceAdapterDefinition[] = [];
  const oauth2ByProvider = new Map<'google' | 'microsoft', SourceAdapterDefinition[]>();

  for (const a of entries) {
    byNamespace.set(a.migrations.namespace, a);
    (byProvider.get(a.provider) ?? byProvider.set(a.provider, []).get(a.provider)!).push(a);
    for (const k of a.capabilities.kinds) {
      (byKind.get(k) ?? byKind.set(k, []).get(k)!).push(a);
    }
    if (a.auth.kind === 'oauth2') {
      oauth2.push(a);
      const p = a.auth.provider;
      (oauth2ByProvider.get(p) ?? oauth2ByProvider.set(p, []).get(p)!).push(a);
    }
  }

  const providers = Object.freeze([...byProvider.keys()]);
  const namespaces = Object.freeze(entries.map((a) => a.migrations.namespace));

  // mutable layer for runtime register/unregister ------------------------
  const overlay = new Map<string, SourceAdapterDefinition>();
  const resolve = (id: string): SourceAdapterDefinition | undefined =>
    overlay.get(id) ?? (adapters as Record<string, SourceAdapterDefinition>)[id];

  // accessors ------------------------------------------------------------
  function isKnownAdapter(v: string): v is AdapterId {
    return resolve(v) !== undefined;
  }
  function assertKnownAdapter(v: string): asserts v is AdapterId {
    if (!isKnownAdapter(v)) {
      throw new CtxindexRegistryError(
        `Unknown adapter "${v}". Known: ${adapterIds.join(', ')}.`,
        'registry_unknown_adapter',
        { adapterId: v, known: adapterIds },
      );
    }
  }
  function getAdapter<TId extends AdapterId>(id: TId): TAdapters[TId] {
    const a = resolve(id) as TAdapters[TId] | undefined;
    if (!a) throw new CtxindexRegistryError(
      `Unknown adapter "${id}".`,
      'registry_unknown_adapter',
      { adapterId: id, known: adapterIds },
    );
    return a;
  }

  // ... full implementations for every method on the handle, all O(1)
  // via the pre-built indexes above. Omitted here for brevity.

  return {
    adapters,
    adapterIds,
    providers,
    namespaces,
    isKnownAdapter,
    assertKnownAdapter,
    getAdapter,
    /* ...rest of handle methods... */
  } as CtxindexAdapterRegistryHandle<TAdapters>;
}
```

## 4. Source adapter operation contract

Adapters should return typed operations instead of writing core tables directly. Adapters may also write adapter-owned namespaced tables through a scoped database handle provided by core inside the sync transaction.

Sketch:

```ts
type SyncOperation =
  | { type: "upsertItem"; item: NormalizedItem }
  | { type: "upsertChunk"; chunk: NormalizedChunk }
  | { type: "tombstoneItem"; ref: ExternalRef; deletedAt: Date }
  | { type: "checkpoint"; checkpoint: unknown; counts?: SyncCounts }
  | { type: "setCursor"; cursor: unknown }
  | { type: "rawRecord"; record: RawRecord };
```

Adapter output should be streamable so large sources do not need to hold a whole mailbox, calendar, or directory walk in memory:

```ts
type SourceAdapter = {
  id: SourceAdapterId;
  sync(ctx: SyncContext): AsyncIterable<SyncOperation> | Promise<SyncResult>;
};
```

This does not eliminate ORM type checking. It splits DB authority by table ownership:

- source adapters are type checked against the operation schema;
- adapter-specific tables are type checked with their own Drizzle schemas;
- adapters may write only tables in their own namespace, through `AdapterDbScope`;
- core validates operations at runtime with schemas before committing;
- core repositories apply core-table operations with Drizzle in the same transaction;
- core advances cursors only after core writes, adapter-scoped writes, and index writes commit.

The adapter/core boundary should be strongly typed, but it should not expose raw core table writes to adapters.

Example boundary:

```ts
type AdapterDbScope<TNamespace extends string> = {
  namespace: TNamespace;
  insert: <TTable extends AdapterOwnedTable<TNamespace>>(table: TTable) => InsertBuilder<TTable>;
  update: <TTable extends AdapterOwnedTable<TNamespace>>(table: TTable) => UpdateBuilder<TTable>;
  delete: <TTable extends AdapterOwnedTable<TNamespace>>(table: TTable) => DeleteBuilder<TTable>;
  executeSql?: (statement: NamespacedSql<TNamespace>) => Promise<unknown>;
};
```

The shared SQLite/Drizzle connection is an implementation detail. Authority is scoped by table namespace and by core-owned invariants.

Adapter-owned tables may reference core tables with foreign keys, but not the other way around:

```ts
export const googleMailThreads = sqliteTable("google_mail_threads", {
  id: text("id").primaryKey(),
  sourceId: text("source_id").notNull().references(() => sources.id, { onDelete: "cascade" }),
  itemId: text("item_id").references(() => items.id, { onDelete: "cascade" }),
  threadId: text("thread_id").notNull(),
});
```

Drizzle should accept this as long as:

- adapter table modules import the core table symbols they reference;
- the migration/schema build includes both core schemas and bundled adapter schemas;
- imports are one-way from adapter schema to core schema to avoid circular imports;
- SQLite foreign keys are enabled for connections;
- FTS/virtual tables are handled separately because they do not behave like normal FK-bearing tables.

If Drizzle migration generation becomes awkward with modular adapter schemas, keep Drizzle table definitions for type checking but let the ctxindex migration coordinator apply generated or custom adapter migrations by namespace.

## 5. Sync run and checkpoint schema

Model sync attempts explicitly instead of hiding them inside source rows.

```text
source_sync_state
  source_id                 PK/FK sources.id
  current_cursor_json       latest committed adapter cursor
  last_successful_run_id    FK sync_runs.id nullable
  last_status               idle | syncing | failed | needs_auth | disabled
  last_started_at
  last_completed_at

sync_runs
  id                        ctxindex-generated UUID/ULID
  source_id                 FK sources.id
  mode                      sync | resync | diff
  status                    running | completed | failed | cancelled
  started_at
  completed_at              nullable
  cursor_before_json        copied from source_sync_state at run start
  cursor_after_json         set only on successful commit
  items_seen
  items_upserted
  items_tombstoned
  chunks_written
  errors_count
  error_summary             nullable, sanitized

sync_run_checkpoints
  id
  run_id                    FK sync_runs.id
  sequence
  checkpoint_json           adapter-owned resume/checkpoint data
  counts_json               optional progress counters
  created_at
```

Execution rule:

1. create `sync_runs(status = running)` before adapter fetch starts;
2. read `source_sync_state.current_cursor_json` into `cursor_before_json`;
3. consume adapter operations, applying core writes and adapter-scoped writes in bounded transactions;
4. write `sync_run_checkpoints` for safe progress points, but do not advance `source_sync_state.current_cursor_json` yet;
5. update FTS/search in the same durable unit as item/chunk changes for v1 lexical search;
6. on success, write `cursor_after_json`, mark the run completed, and advance `source_sync_state.current_cursor_json` in the final transaction;
7. on failure/cancel, mark the run failed/cancelled and keep `source_sync_state.current_cursor_json` unchanged.

`ctxindex status` should read `source_sync_state` for current source health and `sync_runs` for recent history. `ctxindex sync --resume` can later use the last checkpoint from a failed run when the adapter declares that checkpoint format resumable.

## 6. Core identity schema

Use ctxindex-generated IDs for core rows. Provider IDs belong in external reference columns/tables, not as `items.id`.

`external_refs` is one-to-many from items to provider/local identifiers. A single item may have multiple external refs, and sync lookup should use the uniqueness key `(source_id, external_kind, external_id)` to find the owning item.

Examples:

```text
Gmail email item:
  message_id         Gmail API message id
  thread_id          Gmail API thread id
  rfc822_message_id  Message-ID header

Local file item:
  file_path          normalized path under source root
  content_hash       optional identity-adjacent lookup, not primary v1 identity
```

Core identity sketch:

```text
items
  id                  ctxindex-generated UUID/ULID
  source_id           FK sources.id
  kind                mail | calendar_event | file | task | ...
  title
  summary
  occurred_at
  updated_at
  deleted_at          nullable tombstone marker

external_refs
  id
  source_id           FK sources.id
  item_id             FK items.id
  external_kind       message | thread | event | file_path | ...
  external_id         provider/local identifier
  UNIQUE(source_id, external_kind, external_id)

item_chunks
  id                  ctxindex-generated UUID/ULID
  item_id             FK items.id
  source_id           denormalized for filtering
  chunk_index
  content
  content_hash
  UNIQUE(item_id, chunk_index)

item_relations
  id
  source_id
  from_item_id        FK items.id
  to_item_id          FK items.id
  relation_type       thread_member | attachment | recurrence | task_project | ...
  metadata_json       optional
```

Local directory source identity and extraction:

- primary external identity: normalized path relative to source root;
- supported v1 inputs: text-like documents and source code treated as text, including `.txt`, `.md`, `.json`, `.csv`, `.html`, `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.go`, `.rs`, `.java`, `.cs`, `.cpp`, `.c`, `.h`, `.sql`, `.yaml`, `.yml`, and `.toml`;
- PDF can be added in v1 if an extraction dependency is cheap and reliable; otherwise it should be the first follow-up extractor;
- defer `.docx`, images/OCR, audio, and video until the text/code path is stable;
- change detection: content hash, mtime, size, extractor ID, extractor version;
- v1 rename behavior: old path tombstoned, new path inserted;
- later rename detection can join on hash/inode-style metadata when safe.

Local directory ignore behavior:

- support source-level `include` and `exclude` globs;
- respect `.gitignore` by default for directories inside a Git worktree;
- avoid a broad `respect_gitignore = false` default escape hatch because it can accidentally index huge generated or secret-heavy trees;
- support exactly one ctxindex-specific override file at the source root, named `.ctxindexignore` (no `.syncignore` alias);
- use gitignore-style negation rules in `.ctxindexignore` so users can intentionally re-include narrow paths, for example `!docs/generated/**/*.md`;
- apply built-in default ignores before expensive hashing/extraction, including `.git`, `node_modules`, `vendor`, `dist`, `build`, `.cache`, `.next`, `.turbo`, `coverage`, and common binary/package output directories.

Ignore matching uses the **`ignore` npm package** for both `.gitignore` and `.ctxindexignore` (full gitignore semantics including negation). Default ignores and source-level `include`/`exclude` globs are layered into the same matcher. File walking uses **`fdir`** with the matcher as a filter; we may switch later if needed.

File classification uses **`file-type`** (magic-byte sniff) so renamed or extensionless files still classify correctly. Binary skips, oversize skips, and unknown MIME skips are recorded with `skipped_reason` and counted in the sync run.

Content and body hashing uses **SHA-256 via `crypto.subtle` / `node:crypto`** (Bun ships both). Hashes are stored as lowercase hex. No BLAKE3 / xxHash dependency for v1.

Config, provider payload, and CLI flag validation uses **Zod** (`zod` v3). Schemas live next to the code that owns them; parsed config is the source of truth at runtime, not the raw TOML.

Local directory storage should keep file metadata and extracted text, not original file bytes:

```text
local_directory_file_state
  source_id
  item_id
  relative_path
  content_hash
  mtime_ms
  size_bytes
  detected_mime
  extractor_id
  extractor_version
  skipped_reason nullable
```

The searchable "content" in SQLite is the extracted text stored in `item_chunks` and indexed through `chunks_fts`. The original file remains on disk as the canonical byte source.

Extracted text, chunks, FTS rows, and future embeddings are rebuildable index state. A later maintenance command should be able to purge and rebuild them from the filesystem without losing source configuration or file identity metadata.

Skipped files should update sync run counters. Normal `ctxindex status` should show counts by reason; verbose status can list sample paths.

Code chunking should be simple in v1: line/block chunks with language and path metadata. AST-aware or tree-sitter chunking is a later enhancement, not a prerequisite for indexing code.

Tombstones:

- use `deleted_at` rather than hard delete for normal source deletes;
- exclude tombstoned items from search by default;
- retain external refs and enough metadata for `resync`/`diff`;
- add maintenance commands later for compaction/purge.

Chunk/search behavior:

- when item content changes, replace chunks wholesale in v1;
- keep chunk IDs ctxindex-generated;
- enforce `UNIQUE(item_id, chunk_index)`;
- FTS should index item-level metadata separately from chunk body/content;
- search returns items with best matching chunks/snippets;
- future embeddings attach to chunks via `embedding_jobs` and `chunk_embeddings`;
- extractable attachments become separate items linked to the parent by `item_relations`.

## 7. Realms and account model

Realm and account model in v1:

```text
realms
  id                ctxindex-generated id or stable slug, e.g. personal | paxia | uni | global
  label
  is_default        boolean  true for the seeded `global` realm
  created_at

accounts
  id
  provider          google | microsoft | clickup | local | ...
  label             user-facing label, e.g. work-gmail, personal-gmail
  external_user_id  provider user id when known
  created_at

identities/grants/sources reference accounts and realms via FKs:
  account_identities.account_id  FK accounts.id
  grants.account_id              FK accounts.id
  sources.account_id             FK accounts.id
  sources.grant_id               FK grants.id
  sources.realm_id               FK realms.id
```

Multi-account rules:

- a provider can have many accounts (e.g. multiple Gmail accounts);
- an account can have many grants (different scope sets, refresh tokens, secret refs);
- a source binds to one grant, one adapter, and one realm;
- two sources MAY share the same grant/account if the provider allows it; otherwise each source gets its own grant.

Realm rules:

- a `global` realm is created on `ctxindex init` and marked as the default realm;
- new sources fall into `global` unless `--realm <name>` is provided;
- realms are flat in v1, no hierarchy;
- realms can share accounts/grants freely; cross-realm credential isolation is not promised because realms are organizational, not security boundaries;
- search with no realm filter spans all realms;
- search with `--realm a` or `--realm a,b` or repeated `--realm` flags spans those realms plus `global`;
- callers can exclude `global` with `--no-global`, or restrict exactly to a set with `--realm-only`;
- realm membership of an item is derived from its source, so no realm column on `items` is needed in v1.

CLI surface:

```text
ctxindex realm add <name>
ctxindex realm list
ctxindex auth add <provider> [--label <label>]
ctxindex auth list
ctxindex source add <adapter> --account <account> [--realm <realm>] ...
```

```text
  internet_message_id      RFC822 Message-ID header, nullable
  provider_thread_id       provider thread/conversation id, nullable
  subject
  from_json
  to_json
  cc_json
  bcc_json
  sent_at
  received_at
  direction                sent | received | self | unknown
  has_attachments
  attachment_count

mail_bodies
  item_id                  PK/FK items.id
  text_body                extracted plain text, nullable
  html_body                optional sanitized/source HTML, nullable
  body_hash

mail_attachments
  id
  parent_item_id           FK items.id
  child_item_id            FK items.id nullable when extractable child item exists
  filename
  mime_type
  size_bytes
  content_hash nullable
  provider_attachment_id nullable
  extraction_status        not_extractable | pending | extracted | failed

account_identities
  id
  account_id               FK accounts.id
  source_id                FK sources.id nullable
  address_or_identifier
  identity_kind            email | provider_user_id | alias
  confirmed_at nullable
```

External refs for mail should include provider IDs and RFC822 IDs when available:

```text
external_refs examples:
  message              Gmail/Microsoft provider message id
  thread               provider thread/conversation id
  rfc822_message_id    Message-ID header
```

`rfc822_message_id` is not the primary item ID, but it is important for future cross-folder/account duplicate reasoning.

Sent/received classification should use account identities plus provider signals. Examples: Gmail Sent label, Microsoft sent folder/category when available, sender address matching one of the account identities, and provider user ID where exposed. Store the resulting classification in mail-specific metadata; do not make it a core item invariant.

Raw MIME or provider JSON belongs in optional `raw_records`, not in `items`. Default raw retention should remain conservative because mail is sensitive.

Extractable attachments become child `items` linked by `item_relations(relation_type = "attachment")`. Non-extractable attachments stay as `mail_attachments` metadata until an extractor is added.

## 9. Calendar source schema

Calendar adapters should keep core `items` lean and split calendar-specific detail into adapter tables. Recurring events should not be materialized into one item per occurrence.

Item mapping:

- one series item per recurring event, with `kind = calendar_event` and series metadata in adapter tables;
- one item per non-recurring event;
- exception items only when an occurrence is modified or cancelled relative to its series.

Recommended model:

```text
calendar_events
  item_id                  PK/FK items.id
  source_id                FK sources.id
  subject
  description_extracted    extracted plain text used for FTS/chunks
  location
  start_at
  end_at
  timezone
  is_all_day
  organizer_json
  attendees_json
  status                   confirmed | tentative | cancelled
  visibility               default | public | private | confidential
  provider_event_id
  provider_ical_uid        nullable, ICAL UID when exposed
  is_series                boolean

calendar_event_series
  item_id                  PK/FK items.id
  rrule                    serialized iCal RRULE
  rdates_json              extra recurrence dates
  exdates_json             excluded recurrence dates
  series_start_at          DTSTART for the series
  series_end_at            DTEND or duration baseline
  duration_seconds         derived series duration
  has_unbounded_recurrence boolean

calendar_event_exceptions
  id
  series_item_id           FK items.id
  exception_item_id        FK items.id nullable when only a tombstone is needed
  original_start_at        the original occurrence time being overridden
  override_kind            modified | cancelled
  override_metadata_json   optional, e.g. changed subject/location/time
```

Runtime expansion rules:

- expand series within a caller-provided window such as `[from, to]`;
- apply RRULE, RDATE, and EXDATE from `calendar_event_series` to enumerate occurrences;
- replace any occurrence that matches a `calendar_event_exceptions.original_start_at` with the exception item, or skip it if the override is `cancelled`;
- cap the expansion window in CLI/API to avoid accidental unbounded enumeration;
- never persist expanded occurrences as new core items.

Attendees and organizer should be metadata first. Use `attendees_json` and `organizer_json` for v1; defer building first-class contact items.

Cancelled occurrences and series:

- a single cancelled occurrence is stored as a `calendar_event_exceptions(override_kind = "cancelled")` row;
- a fully cancelled series tombstones the series item via `items.deleted_at`;
- search excludes tombstoned series and cancelled exceptions by default; an explicit filter can include them.

Search over calendar should index:

- subject, description, location;
- organizer and attendee display strings or email addresses;
- timezone-normalized start/end for time filters;
- series vs single-event nature so users can filter recurring events.

FTS rows should follow the core pattern: item-level metadata in `items_fts` (title, location, attendee/organizer text, etc.), extracted description in `chunks_fts`. Time-range filters belong in the SQL query layer, not in FTS.

## 10. Search ranking and explainability

Use SQLite FTS5 as the v1 search engine unless implementation evidence forces a different local backend.

Recommended v1 schema shape:

```text
items_fts
  item_id UNINDEXED
  title
  summary
  path
  metadata_text

chunks_fts
  chunk_id UNINDEXED
  item_id UNINDEXED
  content
```

Ranking defaults:

- lexical search is always available and uses BM25 from FTS5;
- item metadata and chunk content are queried separately, then merged into item-level results;
- result rows include the best matching chunk IDs/snippets where available;
- use sanitized FTS query construction rather than passing raw user text directly into `MATCH`;
- support a fallback relaxed query path when strict term matching returns no useful results.

Future vector search:

```text
embedding_jobs
  id
  model_id
  status
  started_at
  completed_at

chunk_embeddings
  chunk_id
  model_id
  dimensions
  vector_blob_or_extension_ref
  embedded_at
  UNIQUE(chunk_id, model_id)
```

Hybrid search should use reciprocal rank fusion by default:

```text
score = sum(1 / (60 + rank_position))
```

Keep `60` as the default RRF constant. Make it an internal setting only if real usage shows a need; do not expose ranking knobs in the first CLI unless debugging requires it.

`ctxindex search --explain` should be added early. It should show, per result:

- matched item ID/source/kind/title;
- whether the result came from item FTS, chunk FTS, vector search, or hybrid merge;
- BM25 rank/score where available;
- vector rank/distance when vector search exists;
- final fused score when hybrid search exists;
- matched chunk IDs and snippets.

Vector or embedding failures must degrade to lexical search, not fail the search command.

## 11. Config, data, cache, logs, and dotfiles

Use XDG Base Directory locations by default:

```text
~/.config/ctxindex/                       # $XDG_CONFIG_HOME
  config.toml
  secret.key                              # mode 0600, when file backend used

~/.local/share/ctxindex/                  # $XDG_DATA_HOME
  ctxindex.sqlite
  secrets.box                             # encrypted file backend, when used
  skills/                                 # bundled skill docs cache, when binary build

~/.local/state/ctxindex/                  # $XDG_STATE_HOME
  logs/
    ctxindex.log
    ctxindex-YYYY-MM-DD.log.gz            # rotated

~/.cache/ctxindex/                        # $XDG_CACHE_HOME
  http/                                   # response cache, purgeable
  extractor/                              # extractor temp
```

`~/.config/ctxindex/` is the dotrepo-friendly home for non-secret persistent configuration. SQLite under `~/.local/share/ctxindex/` stores the local mirror/index and sync state and SHOULD NOT be committed to a dotfiles repo. Logs and state in `~/.local/state/ctxindex/` are operator-facing and SHOULD NOT be committed. Cache content under `~/.cache/ctxindex/` is fully purgeable.

Secrets belong in the OS keychain or encrypted file backend. Config and SQLite store secret references only.

Environment overrides (resolved in this order: explicit `CTXINDEX_*`, then `XDG_*`, then defaults):

```text
CTXINDEX_CONFIG_HOME    -> $XDG_CONFIG_HOME/ctxindex
CTXINDEX_DATA_HOME      -> $XDG_DATA_HOME/ctxindex
CTXINDEX_STATE_HOME     -> $XDG_STATE_HOME/ctxindex
CTXINDEX_CACHE_HOME     -> $XDG_CACHE_HOME/ctxindex
```

A `paths` module exposes `configDir()`, `dataDir()`, `stateDir()`, `cacheDir()`, `logDir()` and is the only legal way to resolve these paths in code. No ad-hoc `~/.local/...` joins. The `paths` module reads env exclusively through the `getEnv()` loader described in §3c — it does not touch `process.env` directly.

A `.env.example` at the repo root documents every `CTXINDEX_*` key with sectioned comments (paths, log, secrets, gmail e2e). Real env files (`.env`, `.env.local`, `.env.*`) are gitignored. The `getEnv()` schema is the source of truth; `.env.example` is regenerated from the schema's metadata or hand-maintained to match.

Source declarations should be hybrid, with TOML as desired state and SQLite as applied state:

- `~/.config/ctxindex/config.toml` stores user-authored non-secret desired config, suitable for dotrepo use after review.
- SQLite stores resolved/applied source records, sync state, cursors, discovered remote metadata, indexes, and run history.
- CLI commands may edit TOML for user-facing configuration and then reconcile/apply it into SQLite.
- Secrets stay outside both TOML and SQLite except for secret references.

## 12. Raw provider payloads

Raw payloads are optional and purgeable.

Prefer one generic raw payload table before adapter-specific debug tables:

```text
raw_records
  id
  source_id
  adapter_id
  external_kind
  external_id
  payload_hash
  payload_json_or_blob
  compression
  fetched_at
  retained_until
```

Default retention should be conservative for sensitive sources such as mail. Candidate defaults:

- raw payload storage off by default; or
- raw payload storage enabled for debug runs only; or
- raw payload storage with a short default TTL.

Adapter-specific tables are for provider-specific query/state that is useful beyond raw debugging, such as Gmail threads, labels, recurring calendar series, or ClickUp custom fields.

## 13. CLI surface and output

### 13.0 End-to-end test harness

Every V1.md §4 exit criterion MUST be exercised by a test that spawns the real `ctxindex` binary (`bun apps/cli/bin/ctxindex.mjs`) inside a sandboxed XDG environment built by `createSandbox()` (§3c). The harness:

1. Allocates a tmpdir per test and sets `CTXINDEX_{CONFIG,DATA,STATE,CACHE}_HOME` to subdirs under it.
2. Spawns the binary with `Bun.spawn` (or equivalent), captures `stdout`/`stderr` and `exitCode`, optionally feeds `stdin`.
3. Wraps long-running commands in `scripts/with-timeout.ts <secs>` so a stuck remote API cannot hang CI; exit `124` is asserted as the timeout case.
4. After the command exits, opens the sandboxed SQLite file with `bun:sqlite` and asserts on row counts in `items`, `chunks`, `attachments`, `sync_runs`, `source_sync_state`, `grants`, `accounts`, `realms`.
5. Greps the sandboxed `logDir()` for any redacted-token regression.

Library-only verifiers (calling `runSync()` / `search()` from TypeScript) are acceptable for unit/integration tests but MUST NOT be the sole evidence for any V1.md §4 criterion. The e2e lane (`bun test:e2e`) holds that contract.

For the Gmail loopback flow, the e2e harness MAY:

- Use mocked OAuth (`MSW`-style fetch interceptor against `oauth2.googleapis.com/token` and `gmail.googleapis.com`) for CI without credentials.
- Use real credentials from `CTXINDEX_GMAIL_CLIENT_ID` + `CTXINDEX_GMAIL_CLIENT_SECRET` + `CTXINDEX_GMAIL_REFRESH_TOKEN` (the e2e harness exchanges the refresh token directly to skip the interactive browser step) when the env vars are present.

The loopback listener implementation lives in `apps/cli/src/auth/google-loopback.ts`: `http.createServer` bound to `127.0.0.1:0`, opens the user's browser via `open` (or platform `xdg-open` / `start`), accepts the first GET to `/callback?code=...`, exchanges, persists, and shuts the listener down. A 5-minute timeout returns exit `50` with an actionable message.

### 13.1 v1 commands

v1 CLI command sketch:

```text
ctxindex init
ctxindex auth add <provider> [--label <label>]
ctxindex auth list
ctxindex realm add <name>
ctxindex realm list
ctxindex source add <adapter> [--account <account>] [--realm <realm>] ...
ctxindex source list
ctxindex source remove <source>
ctxindex sync [<source>] [--mode sync|resync|diff] [--resume]
ctxindex search <query> [--source ...] [--kind ...]
  [--realm <name>]... [--no-global] [--realm-only <name>]...
  [--since ...] [--until ...] [--explain] [--json] [--verbose]
ctxindex status [<source>] [--verbose] [--json]
ctxindex purge index|raw|source <source>
ctxindex skills list
ctxindex skills get <name> [--full] [--json]
ctxindex skills path
```

Output rules:

- default human output is compact, one item per line, only key fields;
- `--verbose` adds extra columns/fields for humans;
- `--json` produces machine-readable JSON; no color, no banners, no prompts;
- detect non-TTY automatically: disable color, progress bars, and decorative output;
- never emit secrets, tokens, raw OAuth refresh values, or full mail bodies in any default output.

Realm filter semantics:

- repeated `--realm` flags and comma-separated values are equivalent;
- when any realm filter is present, the `global` realm is implicitly added to the set;
- `--no-global` removes `global` from an implicit or explicit realm set;
- `--realm-only <name>...` is shorthand for an exclusive realm set with `--no-global`.

Skills surface:

- bundled in a `skills/` directory alongside the binary or inside the published package;
- `ctxindex skills list` prints names and short summaries;
- `ctxindex skills get <name>` prints the primary skill markdown; `--full` inlines referenced docs; `--json` returns structured output;
- `ctxindex skills path` prints the resolved bundled-skills directory;
- bundled skills are versioned with the ctxindex release that ships them; agents should rely on `skills get` rather than reading external mirrors.

Library surface:

- core, registry, sync engine, search, and storage are exposed as library modules so future surfaces such as MCP servers, TUIs, or daemons can be added without rewriting them;
- CLI is one surface on top of the same library; it must not own behavior that other surfaces cannot reach.
